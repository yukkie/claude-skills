# IDD フロー — JS/フロントエンド実装（Phase 3〜6）

JS 側（`frontend/` 配下）の実装 Issue に適用する。

---

## Phase 3 — ドキュメント確認（読む）

以下を順番に読む:
```bash
gh issue view {番号} --repo yukkie/AgentVillage
```
- `doc/Spec.md` — 何を作るか（ゲームルール・機能要件）
- `doc/FrontendDesign.md` — フロントエンド設計（コンポーネント・CSS方針・スタブ差し替え計画）
- Issue に関連する `frontend/src/` 配下の実装ファイル

読み終えたら、以下の形式で**現状報告**をユーザーに提示する。これをスキップして設計提示に進んではならない。

```
## Phase 3 現状確認

### Issue の概要
{Issue のゴールと背景を2〜3文で}

### 既存仕様・設計（ドキュメントから）
- Spec.md: {関連する仕様を箇条書き}
- FrontendDesign.md: {関連するコンポーネント設計・スタブ差し替え方針を箇条書き}

### 既存実装（frontend/src/ から）
- {ファイル名}: {現状の実装の要点を箇条書き}

### 今回の変更方針（ここまで読んで見えてきたこと）
- {変更が必要な箇所と理由を箇条書き}
- {ドキュメントと実装のズレがあれば明記する}
```

この現状報告を提示した後、Phase 4 へ進む。

---

## Phase 4 — 設計フェーズ

Phase 3 の現状報告をもとに設計を整理して提示する:

```
## Issue #{番号}: {タイトル}

### 変更対象ファイル
- `frontend/src/xxx/yyy.jsx` — 〇〇を変更
- `frontend/src/lib/yyy.test.js` — テストを追加

### 変更内容
1. ...
2. ...

### 設計上の判断ポイント
- **判断1**: ...（選択肢A vs 選択肢B → Aを選ぶ理由）
- **判断2**: ...

### 影響範囲
- ...

### 関連 Issue（重複・依存）
設計中に「この変更で別 Issue の AC も満たされる」「この Issue が先決」などの関係に気づいた場合:
- 関係する Issue を列挙する
- 設計承認後、実装前に **各 Issue にコメントで関係性を記載する**（例: `gh issue comment`）

```

設計案を提示したら、承認を待たずに Phase 4.5 へ進む。

---

## Phase 4.5 — 設計改善レビュー

Phase 4 で設計を考えた結果を踏まえ、以下を必ずユーザーに提示して承認を得る。

改善提案がある場合・ない場合どちらも、このフェーズで必ず確認を取る。

### 改善提案がある場合

以下の優先順位で提案するかどうか判断する:

1. **提案する**: 今回の Issue の実装をよりシンプルに実現できる設計変更、またはドキュメントが実装の実態と乖離していて混乱の原因になるもの
2. **提案する（任意）**: Issue に記載されている将来の拡張・変更可能性に関連する設計改善
3. **提案しない**: Issue と無関係な一般的な拡張性・移植性の改善

提示フォーマット:

```
## Phase 4.5 — 設計改善レビュー

### 気づいた課題
- {設計を考えた結果気づいた障害・非効率な点}

### Issueの実装がどう楽になるか
{変更前と変更後の比較を簡潔に}

### 提案する変更
- `doc/FrontendDesign.md` の {セクション}: {変更内容}
（記載漏れの場合は「追記提案」として明示する）

この設計で進めてよいですか？設計変更も先に行いますか？ [y / 変更先に / 修正コメント]
```

### 改善提案がない場合

```
## Phase 4.5 — 設計改善レビュー

既存設計・ドキュメントとの乖離は見当たりませんでした。
Phase 4 の設計のままで進めてよいですか？ [y/N / 修正コメント]
```

**y の場合**: Phase 5 へ進む。
**変更先に の場合**: ドキュメントを更新してから Phase 5 へ進む。
**修正コメントの場合**: 設計を更新して Phase 4.5 を再提示する。

---

## Phase 5 — ドキュメント更新（書く）

設計承認後、以下のドキュメントに今回の変更を反映する:
- `doc/Spec.md` — 機能要件の変更があれば更新
- `doc/FrontendDesign.md` — コンポーネント設計・スタブ差し替え状況を更新

**更新内容をユーザーに確認してから実装へ進む。**
変更がなければ「ドキュメントは最新です。実装に進みます。」と伝えて次へ。

### `doc/GameData.md` の更新基準

フロントエンド実装 Issue では、実装中に発覚した以下の2種類のギャップを `doc/GameData.md` に記録する（Phase 6 実装後、コミット前に追記する）:

**1. データギャップ** — フロントエンドが必要だがログ（`public_log.jsonl` / `agents/*.json`）に存在しないフィールド

| 記録すべき内容 | 例 |
|---|---|
| フィールド名・用途・現状（❌ ログにない等）・対応方針 | `votes`: 観戦upvote数、ログにない、スタブ固定 |

**2. スタブUI** — データとは無関係に、機能が未実装のまま表示されている UI 要素

| 記録すべき内容 | 例 |
|---|---|
| UI要素・現状・対応方針 | ▲▼ upvote ボタン: アクションなし、仕様未定 |

> **判断基準**: 「このUIをユーザーがクリックしたとき何も起きない」「この値は常にスタブ固定」であれば記録対象。
> Playwright でブラウザ確認した際に目視でスタブ UI を洗い出し、漏れなく記録すること。

## Phase 6 — 実装

### コーディングルール

- **データアクセス境界**: fetch URL は専用モジュール（例: `archiveLoader.js`）に集約する。将来 FastAPI エンドポイントへの差し替えがそのファイルの1行変更で済むように設計する
- **Legacy-Adapter**: アーカイブデータの形式互換は `frontend/src/legacy/` に隔離する（Python 側の `src/legacy/` と同じ方針）。`Legacy-Adapter` コメントマーカーを付ける
- **Vite dev server での静的配信**: `state_archive/` などリポジトリルート配下のファイルを配信する場合、`fs.allow` だけでなく `configureServer` ミドルウェアで明示的にルーティングする（`fs.allow` のみでは `index.html` が返されることがある）
- **実装中に既存コードとの一貫性向上やシンプル化に気づいたら、1〜2文で提案してよい**。承認後に実装する

### AC 対応テストチェック

テストを追加する前に、Issue の Acceptance Criteria を1件ずつ確認する:

```bash
gh issue view {番号} --repo yukkie/AgentVillage --json body --jq '.body'
```

各 AC 項目に対して「この条件を検証するテストが存在するか」を照合する。
対応するテストがなければ、実装と合わせてテストを追加する。

すべての AC に対応テストが揃ったら、以下の表をユーザーに提示して承認を得る:

```
## AC 対応テスト確認（最終）

| AC | 対応テスト | 状態 |
|---|---|---|
| {AC 項目} | {テスト関数名（ファイル名::関数名）} | ✅ |
```

> **⚠️ 重要**: この表はカバレッジ突合せの**前**に提示する必須ゲートである。

### Contract テストの TDD ルール

AC 対応テストに contract テストが含まれる場合、そのテストは**実装より先に書き、RED を確認してから実装に入ること**。
これは純粋関数（変換・パース関数など）も例外ではない。

#### 手順

1. contract テストを書く
2. `npm test` で **RED を確認する**（RED にならない場合はアサートが甘い — 見直す）
3. 実装する
4. GREEN を確認する
5. カバレッジ確認へ進む

> **⚠️ 注意**: RED 確認はアサートの正しさを検証する唯一のゲートである。純粋関数であっても同様。

### テスト docstring チェック

新規追加・変更したテスト関数に以下の4要素が揃っているか確認する:

```js
/*
SUT: {テスト対象の関数・クラス・メソッド名}
Mock: {使用するMock/vi.fnとその目的。なければ「なし」}
Level: unit / integration / e2e
Objective: このテストが何を検証するかを1文で記述する
*/
```

欠けている場合は補完してから次へ進む。

### 実装完了後のチェック

```bash
npm test          # vitest run
```

エラーがあれば修正してから次のフェーズへ進む。

Python 側に変更がある場合は合わせて確認:
```bash
ruff check .
pytest --cov=src --cov-report=term-missing
```

#### カバレッジ確認

JS のカバレッジは vitest で計測する（`@vitest/coverage-v8` 導入済みの場合）:

```bash
npm run test:coverage   # vitest run --coverage
```

追加した実装行に対してテストが存在するか確認する。カバーできていない行がある場合:
- まず「このコードは本当に必要か」を問う
- 削除できない場合はテストを追加する
- どうしても対処できない場合は理由をユーザーに説明して承認を得る

以下の表を必ずユーザーに提示し、承認を得てからコミットする:

```
## カバレッジ確認結果

### 追加テスト済み
- {ファイル:行} — {テストした内容}

### カバー不可（承認をお願いします）
| ファイル:行 | 理由 |
|---|---|
| frontend/src/lib/xxx.js:42 | fetch を伴う統合パス（fetch モックは vitest で可能だが Flakyリスクのため省略） |
```

> **⚠️ 重要**: カバレッジ確認結果の表は **PR 作成前**にユーザーに提示する必須ゲートである。

コミットは実装完了・テスト通過・カバレッジ確認（承認済み）後に行う:
```bash
git add {変更ファイルを個別に指定}
git commit -m "{コミットメッセージ}"
```

コミットメッセージの形式: `{type}: {変更内容の要約} (closes #{番号})`
例: `feat(frontend): connect game list to real archives (closes #338)`

# IDD フロー — Python 実装（Phase 3〜6）

Python 側（`src/` 配下）の実装 Issue に適用する。

---

## Phase 3 — ドキュメント確認（読む）

以下を順番に読む:
```bash
gh issue view {番号} --repo yukkie/AgentVillage
```
- `doc/Spec.md` — 何を作るか（ゲームルール・機能要件）
- `doc/DataSpec.md` — データ契約の SSOT（EventType 一覧・可視性ルール・ログ／エージェント JSON スキーマ）。`LogEvent` / `EventType` / エージェント JSON を扱う Issue では必ず参照する
- `doc/Architecture.md` — どう作るか（コンポーネント設計）
- `doc/DetailDesign.md` — どう実装するか（モジュール・クラス詳細）
- Issue に関連する `src/` 配下の実装ファイル

読み終えたら、以下の形式で**現状報告**をユーザーに提示する。これをスキップして設計提示に進んではならない。

```
## Phase 3 現状確認

### Issue の概要
{Issue のゴールと背景を2〜3文で}

### 既存仕様・設計（ドキュメントから）
- Spec.md: {関連する仕様を箇条書き}
- Architecture.md: {関連するコンポーネント設計を箇条書き}
- DetailDesign.md: {関連するクラス・モジュール詳細を箇条書き}

### 既存実装（src/ から）
- {ファイル名}: {現状の実装の要点を箇条書き}

### 今回の変更方針（ここまで読んで見えてきたこと）
- {変更が必要な箇所と理由を箇条書き}
- {ドキュメントと実装のズレがあれば明記する}
```

この現状報告を提示した後、Phase 4 へ進む。

---

## Phase 4 — 設計フェーズ

Phase 3 の現状報告をもとに設計を考える。

設計を考える過程で以下のいずれかに気づいたら、設計提示の前に **Phase 4.5** へ進む:

**コードレベルのスメル**（変更内容から判定）
- **Shotgun Surgery** — 1つの変更が多数ファイルに波及する
- **Divergent Change** — 1つのモジュールが複数の理由で変更される
- **DRY違反** — 同じロジックが複数箇所に出現する
- **デッドコード** — 今回の変更で使われなくなるコードが残る

**設計レベルの問題**（構造を見て判断）
- 低結合・高凝集の違反（変更が波及しすぎる、または責務が混在している）
- ドキュメントと実装の命名・概念の不一致
- 責務の境界が曖昧でテストが書きにくい構造

何も気づかなければそのまま設計を提示してユーザーの承認を得る:

```
## Issue #{番号}: {タイトル}

### 変更対象ファイル
- `src/xxx/yyy.py` — 〇〇を変更
- `tests/test_yyy.py` — テストを追加

### 変更内容
1. ...
2. ...

### 設計上の判断ポイント
- **判断1**: ...（選択肢A vs 選択肢B → Aを選ぶ理由）
- **判断2**: ...

### 影響範囲
- ...

### 関連 Issue（重複・依存）
設計中に「この変更で別 Issue の AC も満たされる」「この Issue が先決」などの関係に気づいた場合:
- 関係する Issue を列挙する
- 設計承認後、実装前に **各 Issue にコメントで関係性を記載する**（例: `gh issue comment`）
  - 依存先 Issue: 「#XX の実装により本 Issue も解消される。PR #YY 参照」
  - 同時クローズされる Issue: 「#XX, #ZZ とともに PR #YY でクローズ。理由: ...」

この設計で実装を進めてよいですか？ [y/N / 修正コメント]
```

承認後は **設計承認後の外部レビュー（任意）** へ進む。修正コメントがあれば設計を更新して再提示する。

### 設計承認後の外部レビュー（任意・shift-left）

設計がユーザーに承認されたら、Phase 5 に進む前に一度だけ問う:

```
この設計で外部レビューを挟みますか？ [y/N]
（挟む場合: 承認済み設計を Issue にコメント投稿し、CODEX / 他 AI / 人間が読める状態にします）
```

- **N** — そのまま Phase 5 へ進む（従来どおり）。
- **y** — 承認済み設計を対象 Issue にコメント投稿し、外部の目に晒す:
  ```bash
  gh issue comment {番号} --repo yukkie/AgentVillage --body "..."
  ```
  本文には Phase 4 で承認された設計（変更対象ファイル・変更内容・判断ポイント・影響範囲）をそのまま載せる。宛先は固定しない — 可視な場所に設計を置くことでレビュー可能にするのが目的。

  レビュー指摘が返ってきたら、その指摘が**当初 AC の範囲内か**を確認する。当初 AC になかった要求が含まれる場合は `SKILL.md` の「**スコープ拡張ゲート（横断）**」へ進む（取り込む / 別 Issue 化 / 却下 をユーザーが選択）。範囲内の設計修正であれば Phase 4 に戻って設計を再提示する。

レビューを挟まない／指摘が片付いたら Phase 5 へ進む。

---

## Phase 4.5 — コードスメル対話（割り込みセッション）

Phase 4 の設計検討中にコードスメルや設計の問題を発見した場合、設計提示を一旦中断してこのセッションに入る。

**役割**: 問題点をユーザーに提示し、一緒に対処方針を決める。改善方法の提案はしない。

```
## Phase 4.5 — 設計の問題点

### 気づいた点
- **{スメルの種類}**: {具体的な箇所と何が問題か}
- **{スメルの種類}**: {具体的な箇所と何が問題か}

どう対処しますか？（改善する / 後回し / このまま進む）
```

ユーザーとの対話で各問題点の方針が決まるまでこのセッションを続ける。すべての問題点に決着がついたら Phase 4 に戻り、決定した方針を踏まえて設計を提示する。

- **改善する** → 必要に応じてドキュメント・コードを先に修正してから Phase 4 へ戻る
- **後回し / このまま進む** → そのまま Phase 4 へ戻る

---

## Phase 5 — ドキュメント更新（書く）

設計承認後、以下のドキュメントに今回の変更を反映する:
- `doc/Spec.md` — 機能要件の変更があれば更新
- `doc/Architecture.md` — コンポーネント設計の変更があれば更新
- `doc/DetailDesign.md` — モジュール・クラス詳細を更新

**更新内容をユーザーに確認してから実装へ進む。**
変更がなければ「ドキュメントは最新です。実装に進みます。」と伝えて次へ。

---

## Phase 6 — 実装

### コーディングルール（CLAUDE.md より）
- インポートは絶対インポート: `from src.xxx import yyy`
- LLM に真実（役職・夜の結果）を持たせない
- LLM 出力は必ず JSON で受け取る（`speech`・`thought`・`intent`・`memory_update`）
- `thought`（腹の中）と `speech`（表の言葉）は別フィールドで管理
- **実運用での再現が難しいパス（エラーハンドリング・エッジケース・レアケース）は、
  優先度に関わらず単体テストを必ずセットで実装する（`⚠️unit test mandatory`）**
- テストの書き方・docstring規約・テストレベル定義は [`tests/TestStrategy.md`](../../../../tests/TestStrategy.md) に従う
- **実装中に既存コードとの一貫性向上やシンプル化に気づいたら、1〜2文で提案してよい**（Phase 4.5 に限らない）。承認後に実装する

### スコープ拡張ゲート（横断）

ユーザーの追加指摘や PR レビュー指摘で**当初 AC になかった要求**が来たら、`SKILL.md` の「スコープ拡張ゲート」へ従う（AC 案提示 → 取り込む/別 Issue 化/却下 をユーザーが選択）。「取り込む」を選んだ場合は **AC 更新 → contract テスト設計（下の TDD ルールで RED 確認）→ 境界ケース列挙** の順で進めてから、下の「AC 対応テストチェック」に戻る。詳細は `SKILL.md` を SSOT とする。

### AC 対応テストチェック

テストを追加する前に、Issue の Acceptance Criteria を1件ずつ確認する:

```bash
gh issue view {番号} --repo yukkie/AgentVillage --json body --jq '.body'
```

各 AC 項目に対して「この条件を検証するテストが存在するか」を照合する。
対応するテストがなければ、実装と合わせてテストを追加する。

```
## AC 対応テスト確認

| AC | 対応テスト | 状態 |
|---|---|---|
| LLM prompts request reasoning field | test_xxx_prompt_includes_reasoning | ✅ |
| Spectator log displays reasoning | test_renderer_vote_shows_reasoning | ❌ 追加が必要 |
```

未対応の AC があればテストを追加する。

すべての AC に対応テストが揃ったら、以下の表をユーザーに提示して承認を得る:

```
## AC 対応テスト確認（最終）

| AC | 対応テスト | 状態 |
|---|---|---|
| {AC 項目} | {テスト関数名（ファイル名::関数名）} | ✅ |
| {AC 項目} | （型定義の削除のみ／間接検証） | ✅ |
```

> **⚠️ 重要**: この表はカバレッジ突合せの**前**に提示する必須ゲートである。
> 表の提示と承認なしにカバレッジ確認・コミット・PR 申請へ進んではならない。

### Contract テストの TDD ルール

AC 対応テストに contract テストが含まれる場合、そのテストは**実装より先に書き、RED を確認してから実装に入ること**。
これは純粋関数（変換・パース関数など）も例外ではない。

#### 手順

1. contract テストを書く
2. **コンパイルエラーレベルの RED を解消する**: ImportError / AttributeError など構文・名前解決エラーで RED になっている場合は、先に空の関数・クラス・フィールドを作って解消する。コンパイルエラーの RED は「テストが壊れているだけ」の false positive であり、アサートの正しさを検証できない。
3. `pytest` で **アサートレベルの RED を確認する**（RED にならない場合はアサートが甘い — 見直す）
4. 実装する
5. GREEN を確認する
6. 通常のカバレッジ確認フロー（`coverage_diff.py`）へ進む

> **⚠️ 注意**: RED 確認はアサートの正しさを検証する唯一のゲートである。
> コンパイルエラーではなくアサートエラーで RED になっていることを必ず確認すること。
> RED を確認せずに GREEN だからといって実装済みと判断してはならない。
> 純粋関数であっても同様。

#### 機能削除時の退行テスト

廃止したハック・旧挙動が残っていないことを確認する退行テスト（例: `assert len(think_events) == 0`）は、contract テストとして**必須**で追加する。ただし GREEN を確認した時点で役目は終わるため、**GREEN 確認後すぐに削除する**。残し続けると意図不明なテストが蓄積する技術的負債になる。

### テスト docstring チェック

新規追加・変更したテスト関数に以下の4要素が揃っているか確認する:

```python
"""
SUT: {テスト対象の関数・クラス・メソッド名}
Mock: {使用するMock/monkeypatchとその目的。なければ「なし」}
Level: unit / integration / e2e
Objective: このテストが何を検証するかを1文で記述する
"""
```

欠けている場合は補完してから次へ進む。

### 実装完了後のチェック
```bash
ruff check .
pytest --cov=src --cov-report=term-missing
```

エラーがあれば修正してから次のフェーズへ進む。

#### カバレッジ突合せ

`pytest` 通過後、カバレッジレポートの `Missing` 行と今回の変更差分を突合せる。
補助スクリプト `~/.claude/skills/idd/scripts/coverage_diff.py` を使う（表を自動生成してくれる）:

```bash
SCRIPT="$HOME/.claude/skills/idd/scripts/coverage_diff.py"

# coverage.json を生成（まだなければ）
pytest --cov=src --cov-report=json -q

# stage 前（未コミットの変更を確認）
python "$SCRIPT" --unstaged

# stage 後 / コミット後（master との差分を確認）
python "$SCRIPT"
```

スクリプトは git diff の各追加行（`+` で始まる）にカバレッジマーカーを付けて出力する:

| マーカー | 意味 |
|---|---|
| `[o]` | カバー済み（テスト不要） |
| `[x]` | Missing（テスト追加が必要） |
| `[ ]` | 計測対象外（コメント・空行・型アノテーション等） |

`[x]` が付いた行がある場合はテストを追加してから再度チェックする。差分に含まれる追加行（`+` で始まる実装行）が `[x]` のまま残っていないか確認する。

- **Missing が残っている場合**:
  - **まず「このコードは本当に必要か」を問う**: 実運用上到達しないコードを発見した場合、カバレッジ対象外として扱ってはならない。削除・設計変更・`raise` への置き換えのいずれかを先に提案すること。削除できる場合はカバレッジ問題ごと解消する（`tests/TestStrategy.md §7-A` 参照）。
  - 削除できない場合、「設計上このパスはテスト可能なはずでは？」と Phase 4 の設計に立ち返って確認する。設計を見ても判断できない場合は該当コードを読み、テスト可能な入力経路がないか確認する（LLM・ユーザー入力起因の外部境界フォールバックはテストを書ける — `tests/TestStrategy.md §7-B` 参照）。
  - テストを追加できる場合: 実装してから再度チェックする
  - どうしても追加テストで対処できない場合: `tests/TestStrategy.md §8 意図的未カバー領域` に類例がないか確認する。
    該当する場合は表の「理由」欄に §8 の該当行を明記する（例: `tests/TestStrategy.md §8 — 実LLM API呼び出し`）。
    該当しない場合は、なぜテスト不可かを自力で説明してからユーザーに承認を求める。

以下の表を必ずユーザーに提示し、承認を得てからコミットする:

```
## カバレッジ確認結果

### 追加テスト済み
- {ファイル:行} — {テストした内容}

### カバー不可（承認をお願いします）
| ファイル:行 | 理由 |
|---|---|
| src/xxx.py:42 | tests/TestStrategy.md §8 — 実LLM API呼び出し（APIコスト・Flakyリスクのため MagicMock で代替） |
| src/yyy.py:87 | OS依存の例外パスで再現困難（§8 非該当 — 手動確認済み） |
```

> **⚠️ 重要**: カバレッジ確認結果の表は **PR 作成前**にユーザーに提示する必須ゲートである。
> `pytest` 通過後にすぐコミット・PR 申請してはならない。カバレッジ確認 → 承認 → コミット の順序を守ること。

コミットは実装完了・テスト通過・カバレッジ確認（承認済み）後に行う:
```bash
git add {変更ファイルを個別に指定}
git commit -m "{コミットメッセージ}"
```

コミットメッセージの形式: `{type}: {変更内容の要約} (closes #{番号})`
例: `feat: add pre-night judgment phase for day 2+ (closes #21)`

# Issue Driven Development (IDD) — 実装フロー

このスキルは AgentVillage の**実装フロー**（既存 Issue → 実装 → PR）をガイドする。

> **役割分担**: 新規 Issue 作成・既存 Issue の revise・メモの import は `/issue`。バックログ refinement（昇格・優先度・SP）は `/backlog`。本スキルは「Issue を選んで実装し PR を出す」ことに専念する。

## 通常フロー

Issue選択 → ブランチ作成 → ドキュメント確認 → 設計承認 → ドキュメント更新 → 実装 → PR作成 の順に進み、**フェーズが変わるたびにユーザーの確認を取る**。

## フェーズ概要

```
[1] Issue 選択
[2] ブランチ作成              ← 確認
[3] ドキュメント確認（読む）
[4] 設計検討 → スメル発見で 4.5 へ中断、なければ設計提示 → 承認後に任意で外部レビュー ← 確認
[4.5] コードスメル対話        ← 決着まで対話ループ → 4 に戻る
[5] ドキュメント更新（書く）
[6] 実装 + テスト
[7] PR 作成                   ← 確認
[8] 教訓の振り返り（任意・pull 型） ← PR 後の区切りで一度だけ。y/N

[*] スコープ拡張ゲート（横断）  ← 任意フェーズで発火。ユーザー追加指摘 / PR レビュー指摘で当初 AC 外の要求が来たとき
```

## Phase 1 — Issue 選択

### 番号が指定された場合（例: `/idd 42`）
そのまま Phase 2 へ進む。

### 番号が指定されていない場合
`doc/Ideas.md` を読み込み、未実装 Issue の一覧を表示する。

表示フォーマット:
```
## 取り組む Issue を選んでください

| # | 優先度 | ラベル | タイトル | 内容 |
|---|---|---|---|---|
| 37 | 🟡 | tech-debt | `build_system_prompt` のパラメータ過多 | パラメータを整理してシンプルに |
| 21 | （なし） | enhancement | Day 2+ pre-night judgment phase | 昼開始前の判断フェーズを Day 2+ にも拡張 |
...

番号を入力してください（例: 42）:
```

tech-debt は表の先頭に並べる（同一優先度内でも tech-debt を上位に表示する）。

ユーザーが番号を入力するまで待つ。

> **注意**: Ideas.md に載っている Issue が GitHub 上ですでに CLOSED になっている場合は、一覧表示後に「#XX は GitHub 上でクローズ済みです。Ideas.md から削除しますか？」と確認し、承認されたら `docs/cleanup-ideas` ブランチを作成して削除・コミットし、PR を作成する。

## Phase 2 — ブランチ作成

### ブランチ名の決め方

Issue のラベルに応じてプレフィックスを使い分ける:

| ラベル | プレフィックス |
|---|---|
| `enhancement` / その他 | `feature/` |
| `bug` | `bugfix/` |
| `tech-debt` | `refactor/` |

形式: `{prefix}issue-{番号}-{Issue タイトルを kebab-case に変換した短縮形}`

例:
- Issue #37「`build_system_prompt` のパラメータ過多」（tech-debt）→ `refactor/issue-37-build-system-prompt-params`
- Issue #24「Role class refactoring」（enhancement）→ `feature/issue-24-role-class-refactoring`

タイトルが長い場合は 4〜5 単語に収める。

### ブランチ作成手順

#### オープン PR チェック（必須）

ブランチ作成前に、マージされていない PR が存在するか確認する:
```
mcp__plugin_github_github__list_pull_requests(
  owner="yukkie",
  repo="AgentVillage",
  state="open",
)
```

オープン PR が1件以上ある場合は、以下を表示して警告する:
```
⚠️  マージされていない PR があります:

| # | タイトル | ブランチ |
|---|---|---|
| {番号} | {タイトル} | {headRefName} → {baseRefName} |
...

これらが今回の実装と関連している場合、先にマージしておかないとコンフリクトが発生する可能性があります。
続けてブランチを作成しますか？ [y/N]
```

`N` の場合は処理を中断する。`y` の場合は次の手順へ進む。

#### 現在のブランチが `master` の場合

確認なしで以下を実行する:
```bash
git pull --rebase origin master
git checkout -b {prefix}issue-{番号}-{名前}
```

#### 現在のブランチが `master` 以外の場合

以下を提示してユーザーに確認する:
```
現在のブランチ: {現在のブランチ}

作業ブランチを作成するには master ベースが必要です。
master に切り替えて pull しますか？ [y/N]
```

承認後:
```bash
git checkout master
git pull --rebase origin master
git checkout -b {prefix}issue-{番号}-{名前}
```

拒否された場合は処理を中断し、ユーザーに手動で master へ切り替えるよう案内する。

## Phase 3〜6 — 実装対象による分岐

### 事前調査（設計 Issue・横断的変更では必須）

Issue が **設計 Issue（doc 更新・スキーマ定義）** または **横断的変更（ログ/イベントフォーマット・共通データ型・複数モジュールにまたがる契約変更）** の場合、Phase 3 に入る前に以下を**必ず**実施してから設計を組む。これを飛ばすと「見落とし → 設計やり直し」になりやすい（実例: #344 で wolf_chat の同種ハック・CLI renderer の consumer・co_announcement の二重化を初回に拾えなかった）。

1. **関連の全数洗い出し** — 対象 Issue の本文を `mcp__plugin_github_github__issue_read(method="get", ...)` で、コメントを `method="get_comments"` で読む。本文・コメント中の `依存:` / `#NN` 言及をたどって関連 Issue を引く。`memory/MEMORY.md` と関連 memory を確認する。同種キーワードで既存 Issue を検索する（`mcp__plugin_github_github__list_issues` または `mcp__plugin_github_github__search_issues` 相当）。
2. **同種ハック/回避策の全数探索** — 回避策（プレフィックス・別イベント emit・フラグ分岐等）を1つ見つけたら、「同じ回避策が他の経路にもないか」を grep で網羅する。1箇所直して他を残すと中途半端になる。
3. **producer / consumer の全列挙** — スキーマ・フォーマットを変える場合、それを **emit する側（producer）** と **読む側（consumer）** を機械的に全部挙げる。consumer は CLI renderer・JS parser・replay・テストなど複数言語にまたがることが多い（CLI renderer の見落としに注意）。`LogEvent` / `EventType` / `spectator_log.jsonl` / エージェント JSON のデータ契約は `doc/DataSpec.md` が SSOT。EventType 一覧・可視性ルール・スキーマはまずここを参照し、変更時は `doc/DataSpec.md` も併せて更新する。
   - **⚠️ grep が届かない境界に注意** — consumer の中には**プロジェクト外（別リポジトリ）にあるもの**が存在する。とくに idd スキル定義（`~/.claude/skills/idd/` 配下の `SKILL.md` / `references/flow-*.md`）はプロジェクトの doc・データ契約を参照しているが、`d:\git\AgentVillage` を grep しても出てこない。**doc・データ契約・データスキーマを変更／新設するときは、スキル定義側に追従が必要な参照（読み物リスト・事前調査手順・ドキュメント表など）がないかを別途・明示的に確認する。** プロジェクトの変更が別リポジトリの手順書を陳腐化させても、同じ PR では担保されないため、ここで意識的に拾う（実例: #415 で `doc/DataSpec.md` 新設時、Issue 本文は `flow-js.md` のみ挙げていたが、実際は `flow-python.md` / `SKILL.md` / `CLAUDE.md` のドキュメント表も追従が必要だった）。
4. **後方互換の定型確認** — 過去ログ・既存データへの影響、read/write の非対称（emit は新方式・read はフォールバック等）、フォールバックの要否をユーザーに確認する。
5. **分割の選択肢提示** — 実装が大きい場合、**分割の軸を明示して2〜3案**を提示し、判断はユーザーに委ねる（producer/consumer 軸・言語軸など）。AI が案を勝手に1つに決めない。

### 分割提案の必須判定（事前調査の対象外 Issue にも適用）

上の「事前調査」は設計 Issue・横断的変更が対象だが、**分割提案だけは全 Issue に適用する**。Phase 3 で AC を読んだ直後、**AC に含まれる独立変更項目を列挙してカウントする**。独立項目が **5 個以上**、または「再利用可能なコンポーネント／共通化の切り出し」が1件でも含まれる場合は、設計提示の前に**分割提案を必須**とする:

```
## 分割提案（AC 項目数 {N} 個）

このIssueのACには独立した変更が {N} 個あります。1スプリントで作り切るには大きく、
実装中に項目が抜け落ちるリスクがあります。以下の分割案を検討してください:

### 分割案
- **案A（最小遷移に絞る）**: {コア要件だけを今回の Issue に残し、残りは別 Issue 化}
- **案B（重い切り出しを先行）**: {再利用コンポーネント等を先に別 Issue で実装してから本体}
- **案C（このまま1つで進める）**: {分割せず全項目を本 Issue で実装}

どうしますか？（案を選ぶ / このまま進む / 修正コメント）
```

> **分割を実行するときは `/issue split <番号>` を案内する**。分割は子 Issue を作るだけでなく、**分割対象に依存していた既存 Issue の依存リンクを張り替える**必要がある（依存の双方向再配線）。この張り替えは `issue split` サブコマンドが手順として担保する。本フローから直接 `issue_write` で分割しない（張り替え漏れを防ぐ）。

> **核心**: AI は「見かけの SP が小さいから1つでやれる」と判断しがち。だが**独立変更が多いIssueは実装中に視野が狭まり項目が抜け落ちる**（実例: #342「React Router導入」は AC に独立変更が約10項目あり、roster の `Link` 化AC を実装し忘れたまま PR を出し、レビューで初めて発覚した）。規模の最終判断はユーザーに委ねるが、**項目数が閾値を超えたら分割提案そのものはスキップしてはならない**。

### 未解決事項の行き先 — memory ではなく Issue

調査・設計中に見つかった **未解決の作業項目・設計判断・「後でやる」事項** は、その場で解決しないなら**必ず GitHub Issue に反映する**（既存 Issue に追記、なければ新規作成）。**memory には書かない。**

> **フォローアップ Issue を新規作成する場合は `/issue create` フロー（C-1〜C-3）で行う。** `issue_write` を直接呼ぶのは禁止。`/issue create` の C-3 に Ideas.md 追記と PR 作成が含まれており、これを省くと追記漏れになる。
>
> **依存の配線を忘れない**: フォローアップ Issue は元の Issue・PR・共通部品への依存を持つ。`/issue` の「横断ルール: Issue 間の依存を Requirement / AC に書き戻す」に従い、本文冒頭の `依存: #NN`（元 Issue・PR へのリンク）・Requirement の前提・AC の検証条件を必ず書く。これを省くと、後から読む人・別セッションが元の文脈を辿れず、独自実装へドリフトする。

- 理由: Issue 管理の目的（追跡可能・人間も他セッションも追える・作業の単一ソース）は本スキルの前提。未解決事項を memory に入れると検索性が低く分散し、次の作業で recall されず見落とす（実例: #344 で co_announcement 統合の必要性が memory に埋もれ、初回調査で拾えなかった）。
- memory に書いてよいのは「明日この作業の続きをやる」のような**作業途中のセッション状態**だけ。未解決の*仕様・設計・タスク*は Issue へ。
- ユーザーの恒久的な好み・進め方（feedback）は従来どおり memory でよい（それは作業項目ではない）。

### 参照ファイルの読み込み

以下の Phase 3〜6 は**言語非依存の骨格**を本セクションに定義する。**読むドキュメント・コーディングルール・ツールコマンド・言語固有の手順は言語別 references に分離**してある。Phase 3 に入る前に、Issue の変更対象に応じて該当 references を読み、各フェーズの「**→ 言語別**」マーカーでその内容を差し込む:

- **Python 実装**（`src/` 配下）: [`references/flow-python.md`](references/flow-python.md)
- **JS/フロントエンド実装**（`frontend/` 配下）: [`references/flow-js.md`](references/flow-js.md)

両方にまたがる場合は両方を読む。

## Phase 3 — ドキュメント確認（読む）

対象 Issue を読む（`mcp__plugin_github_github__issue_read(method="get", owner="yukkie", repo="AgentVillage", issue_number={番号})`）。続けて、**→ 言語別 references の「Phase 3 読むドキュメント」**に挙げられた doc と、Issue に関連する実装ファイルを順に読む。

読み終えたら、以下の形式で**現状報告**をユーザーに提示する。これをスキップして設計提示に進んではならない:

```
## Phase 3 現状確認

### Issue の概要
{Issue のゴールと背景を2〜3文で}

### 既存仕様・設計（ドキュメントから）
- {doc 名}: {関連する仕様・設計を箇条書き}

### 既存実装（コードから）
- {ファイル名}: {現状の実装の要点を箇条書き}

### 今回の変更方針（ここまで読んで見えてきたこと）
- {変更が必要な箇所と理由を箇条書き}
- {ドキュメントと実装のズレがあれば明記する}
```

この現状報告を提示した後、Phase 4 へ進む。

## Phase 4 — 設計フェーズ

Phase 3 の現状報告をもとに設計を考える。設計を考える過程で以下のいずれかに気づいたら、設計提示の前に **Phase 4.5** へ進む:

**コードレベルのスメル**（変更内容から判定）
- **Shotgun Surgery** — 1つの変更が多数ファイルに波及する
- **Divergent Change** — 1つのモジュールが複数の理由で変更される
- **DRY違反** — 同じロジックが複数箇所に出現する
- **デッドコード** — 今回の変更で使われなくなるコードが残る

**設計レベルの問題**（構造を見て判断）
- 低結合・高凝集の違反（変更が波及しすぎる、または責務が混在している）
- ドキュメントと実装の命名・概念の不一致
- 責務の境界が曖昧でテストが書きにくい構造

何も気づかなければそのまま設計を提示してユーザーの承認を得る:

```
## Issue #{番号}: {タイトル}

### 変更対象ファイル
- `{パス}` — 〇〇を変更
- `{テストパス}` — テストを追加

### 変更内容
1. ...
2. ...

### 設計上の判断ポイント
- **判断1**: ...（選択肢A vs 選択肢B → Aを選ぶ理由）
- **判断2**: ...

> **⚠️ ドリフト確認（整理・構造化・refactor Issue）**: Issue が既存の何か（フィールド・構造・関数）の**整理・構造化**を求めている場合、設計はその対象そのものを変えること。対象を温存したまま別の箱（新フィールド・新イベント・新クラス）を足す設計になったら、それは Issue の目的からのドリフト＝逃げを疑う。整理対象を避けて新設し、旧来のものを「後方互換のため残す」と称してデッドデータ化するのは典型的な回避パターン。
> どうしても対象を直接整理できず新設が必要なら、**Issue の目的から逸脱していることを明示したうえでユーザーの承認を得てから進む**。黙って新設に流さない。

### 影響範囲
- ...

### AC ↔ Contract テスト対応（設計）

Issue の AC を**全行転記**し、各 AC に対応する contract テストを設計段階で確定する
（AC に通し番号 `AC-n:` が付いている場合は番号ごと転記し、以後この番号で参照する）:

| AC | テスト名（識別子） | ファイル | テスト観点・実装ヒント |
|---|---|---|---|
| AC-1: {AC 項目} | {種別}: {識別子} | {テストファイル} | {難所のみ: モック境界・テスト環境の制約等} |
| AC-2: {AC 項目} | {種別}: {識別子} | {テストファイル} | 組み合わせ境界: {交差する既存 state} |
| AC-3: {AC 項目} | {識別子}（GREEN 後削除する退行テスト） | {テストファイル} | |
| AC-4: {AC 項目} | （contract テスト不可） | — | {理由＋代替の担保手段} |

### 関連 Issue（重複・依存）
設計中に「この変更で別 Issue の AC も満たされる」「この Issue が先決」などの関係に気づいた場合:
- 関係する Issue を列挙する
- 設計承認後、実装前に **各 Issue にコメントで関係性を記載する**（`mcp__plugin_github_github__add_issue_comment` ツール）
  - 依存先 Issue: 「#XX の実装により本 Issue も解消される。PR #YY 参照」
  - 同時クローズされる Issue: 「#XX, #ZZ とともに PR #YY でクローズ。理由: ...」

この設計で実装を進めてよいですか？ [y/N / 修正コメント]
```

> **→ 言語別**: JS/フロントエンド実装では、HTML 要素種別を変える際の「暗黙の挙動・スタイル変更」を設計に明記する（`references/flow-js.md`「Phase 4 暗黙の挙動・スタイル変更」参照）。

> **テスト名は識別子で書く（AC ↔ Contract テスト対応）**: 実装にそのまま現れるテスト識別子を書く。テスト目的の言い換えは不可（目的はテスト docstring の `Objective:` が担保する）。この識別子は Phase 6「AC 対応テストチェック」の確認表と突合する。実装中に識別子を変える場合は黙って変えず、Phase 6 の表で設計時識別子との対応を明示する。
> - 識別子の記法と `{種別}:` プレフィックスの類別は**言語別 references の「Phase 4 AC ↔ Contract テスト対応」**を参照
> - 新 AC が既存 state・既存 AC と交差する点を検証するテストには「組み合わせ境界」と明記する
> - 機能削除の退行テストは「GREEN 後削除」と明記する（恒久テストにしない。Phase 6「機能削除時の退行テスト」参照）
> - **contract テストを作れない AC は、作れない理由と代替の担保手段を必ず書く**。典型例（実データを使う frontend テスト等）は言語別 references を参照

承認後は **AC 追従チェック** → **設計承認後の外部レビュー（任意）** の順に進む。修正コメントがあれば設計を更新して再提示する。

### AC 追従チェック（実装方針が AC の表記とズレたら）

設計を詰める過程で**実装方針が Issue の AC が前提していた表記・構造とズレた**場合（例: フィールドの持ち方を `extra_data.x` から top-level `x` に変えた、データ構造を別案に切り替えた）、その AC は実装の検証正本（SSOT）として食い違ったままになる。pr-review で AC を1件ずつ照合できなくなるため、**実装に入る前に AC を新方針ベースへ更新する**。

手順:

1. ズレている AC 項目を列挙し、**新方針ベースの AC 文面（案）**をユーザーに提示する。
2. **OK をもらってから** `mcp__plugin_github_github__issue_write(method="update", owner="yukkie", repo="AgentVillage", issue_number={番号}, body="...")` で Issue 本文の該当 AC を更新する。AC が参照する doc 更新指示（`doc/DataSpec.md` の表記など）もあわせて新方針に揃える。
3. 更新後の AC で Phase 6 の「AC 対応テストチェック」を行う。

> **スコープ拡張ゲートとの違い**: スコープ拡張ゲートは「AC に**新しい受け入れ条件を足す**」（外部から増えた要求）。本チェックは「既存 AC の**表記を実装方針に追従させる**」（要求は不変、表現だけ更新）。要求そのものが増減する場合はスコープ拡張ゲートへ回す。

### 設計承認後の外部レビュー（任意・shift-left）

設計がユーザーに承認されたら、Phase 5 に進む前に一度だけ問う:

```
この設計で外部レビューを挟みますか？ [y/N]
（挟む場合: 承認済み設計を Issue にコメント投稿し、CODEX / 他 AI / 人間が読める状態にします）
```

- **N** — そのまま Phase 5 へ進む（従来どおり）。
- **y** — 承認済み設計を対象 Issue にコメント投稿し、外部の目に晒す（長文 body をシェルに渡さないため `mcp__plugin_github_github__add_issue_comment` ツールを使う）:
  ```
  mcp__plugin_github_github__add_issue_comment(owner="yukkie", repo="AgentVillage", issue_number={番号}, body="...")
  ```
  本文には Phase 4 で承認された設計（変更対象ファイル・変更内容・判断ポイント・影響範囲）をそのまま載せる。宛先は固定しない — 可視な場所に設計を置くことでレビュー可能にするのが目的。

  レビュー指摘が返ってきたら、その指摘が**当初 AC の範囲内か**を確認する。当初 AC になかった要求が含まれる場合は「**スコープ拡張ゲート（横断）**」へ進む。範囲内の設計修正であれば Phase 4 に戻って設計を再提示する。

レビューを挟まない／指摘が片付いたら Phase 5 へ進む。

## Phase 4.5 — コードスメル対話（割り込みセッション）

Phase 4 の設計検討中にコードスメルや設計の問題を発見した場合、設計提示を一旦中断してこのセッションに入る。

**役割**: 問題点をユーザーに提示し、一緒に対処方針を決める。改善方法の提案はしない。

```
## Phase 4.5 — 設計の問題点

### 気づいた点
- **{スメルの種類}**: {具体的な箇所と何が問題か}
- **{スメルの種類}**: {具体的な箇所と何が問題か}

どう対処しますか？（改善する / 後回し / このまま進む）
```

ユーザーとの対話で各問題点の方針が決まるまでこのセッションを続ける。すべての問題点に決着がついたら Phase 4 に戻り、決定した方針を踏まえて設計を提示する。

- **改善する** → 必要に応じてドキュメント・コードを先に修正してから Phase 4 へ戻る
- **後回し / このまま進む** → そのまま Phase 4 へ戻る

**更新内容をユーザーに確認してから実装へ進む。**

## Phase 6 — 実装

### コーディングルール
共通:
- **実装中に既存コードとの一貫性向上やシンプル化に気づいたら、1〜2文で提案してよい**（Phase 4.5 に限らない）。承認後に実装する

> **→ 言語別**: 言語固有のコーディングルール（絶対 import・フィールド名整合・Legacy-Adapter 等）は references を参照。

### 設計フェーズへの差し戻し（実装中に設計の不備を発見したとき）

実装に入ってから、**承認済み設計そのものが不十分・実装不能・誤りだと判明する**ことがある。例:

- 設計が前提していたデータ構造・関数シグネチャが実際には存在せず、設計どおりに書けない
- 設計が見落としていた consumer / producer が実装中に見つかり、設計の変更対象ファイルでは足りない
- AC を満たすには設計とは別のアプローチが必要だと、コードに触れて初めて分かった

このとき**設計の穴を実装で力技で埋めてはならない**。それは Phase 4 で承認された設計とコードが乖離した状態を生み、pr-review で設計と実装を照合できなくなる。**Phase 4（設計フェーズ）へ差し戻す**:

1. 何が・なぜ設計どおりに実装できないかを具体的に説明する（設計のどの前提が崩れたか）。
2. ユーザーに差し戻しを提示する:
   ```
   ## 設計フェーズへの差し戻し

   実装中に、承認済み設計では対応できない点が判明しました。

   ### 設計の崩れた前提
   - {設計が前提していたこと} → {実際に判明したこと}

   ### 影響
   - {このまま実装を続けると何が問題か}

   設計を見直しますか？（設計し直す / 設計を一部修正して続行 / このまま進める理由がある）
   ```
3. ユーザーの判断で Phase 4 に戻って設計を再提示・再承認してから、Phase 5/6 をやり直す。

> **スコープ拡張ゲート・Phase 4.5 との違い**: スコープ拡張は「外部から新しい AC が増えた」、Phase 4.5 は「設計検討中にスメルに気づいた」。本差し戻しは「**実装者が、承認済み設計そのものの不備を実装中に発見した**」— 起点が実装フェーズで、要求は不変だが設計が誤っていたケース。力技で握りつぶさず設計に戻すことで、設計と実装の整合（＝検証可能性）を保つ。

### スコープ拡張ゲート（横断）

ユーザーの追加指摘や PR レビュー指摘で**当初 AC になかった要求**が来たら、本 SKILL.md の「スコープ拡張ゲート（横断）」へ従う（AC 案提示 → 取り込む/別 Issue 化/却下 をユーザーが選択）。「取り込む」を選んだ場合は **AC 更新 → contract テスト設計（下の TDD ルールで RED 確認）→ 境界ケース列挙** の順で進めてから、下の「AC 対応テストチェック」に戻る。

### AC 対応テストチェック

テストを追加する前に、Issue の Acceptance Criteria を1件ずつ確認する:

```
mcp__plugin_github_github__issue_read(
  method="get",
  owner="yukkie",
  repo="AgentVillage",
  issue_number={番号},
)
```

**まず Issue 本文から AC を全項目そのまま転記する**（書いたコードからではなく、AC を出発点にする）。各 AC 項目に対して「**この AC を満たす実装がどのファイル:行に存在するか**」と「**この条件を検証するテストが存在するか**」の両方を照合する。対応する実装・テストがなければ、実装と合わせて追加する。

> **⚠️ 照合の向きを間違えない**: 「書いたコード → テストがあるか」ではなく「**AC全項目 → 実装とテストがあるか**」で照合する。前者だと*実装し忘れた AC は表に現れず見落とす*（実例: #342 で roster の `Link` 化AC を実装も転記もせず、表に行が立たないまま PR を出した）。AC を機械的に全行転記してから埋めること。

すべての AC に対応実装・テストが揃ったら、以下の表をユーザーに提示して承認を得る:

```
## AC 対応テスト確認（最終）

| AC | 対応実装（ファイル:行） | 対応テスト | 状態 |
|---|---|---|---|
| AC-1: {AC 項目} | {実装ファイル:行} | {テスト関数名（ファイル名::関数名）} | ✅ |
| AC-2: {AC 項目} | {実装ファイル:行} | （型定義の削除のみ／間接検証） | ✅ |
```

> **Phase 4 の設計表と突合する**: Phase 4 で「AC ↔ Contract テスト対応（設計）」を提示済みの場合、本表のテスト関数名を設計時の識別子と突合する。識別子を変更した場合は対応関係と理由を表に明記する。AC に通し番号がある場合は本表でも `AC-n:` で参照する（番号なしの旧 Issue は AC 文面で照合する）。

> **⚠️ 重要**: この表はカバレッジ突合せの**前**に提示する必須ゲートである。
> 表の提示と承認なしにカバレッジ確認・コミット・PR 申請へ進んではならない。
> **「対応実装」列が空欄になる AC が1行でもあれば、それは未実装の AC である**。コミットに進まず実装を追加すること。

### Contract テストの TDD ルール

AC 対応テストに contract テストが含まれる場合、そのテストは**実装より先に書き、アサートレベルの RED を確認してから実装に入ること**。これは純粋関数（変換・パース関数など）も例外ではない。

**核心**: RED 確認はアサートの正しさを検証する唯一のゲート。**コンパイル/構文エラーではなくアサートエラーで RED になっている**ことを必ず確認する。コンパイルエラーの RED は「テストが壊れているだけ」の false positive であり、アサートの正しさを検証できない。RED にならない場合はアサートが甘い — 見直す。GREEN だからといって RED 確認を飛ばして実装済みと判断してはならない。

基本手順:
1. contract テストを書く
2. テストを実行し、**アサートレベルの RED を確認する**
3. 実装する
4. GREEN を確認する
5. カバレッジ確認へ進む

> **→ 言語別**: テスト実行コマンドと、構文・名前解決エラー（ImportError 等）で RED になった場合の前処理は references を参照。

#### 機能削除時の退行テスト

廃止したハック・旧挙動が残っていないことを確認する退行テスト（例: `assert len(think_events) == 0`）は、contract テストとして**必須**で追加する。ただし GREEN を確認した時点で役目は終わるため、**GREEN 確認後すぐに削除する**。残し続けると意図不明なテストが蓄積する技術的負債になる。

### テスト docstring チェック

新規追加・変更したテスト関数に以下の4要素が揃っているか確認する（記法は言語のコメント構文に合わせる）:

```
SUT: {テスト対象の関数・クラス・メソッド名}
Mock: {使用する Mock とその目的。なければ「なし」}
Level: unit / integration / e2e
Objective: このテストが何を検証するかを1文で記述する
```

欠けている場合は補完してから次へ進む。

### 実装完了後のチェック

**→ 言語別 references の「Phase 6 実装後チェック」**に従い、リント・テスト・カバレッジ計測を実行する（JS はさらに Playwright によるブラウザ動作確認が必須ゲート）。エラーがあれば修正してから次へ進む。

#### カバレッジ確認結果（共通の必須ゲート）

追加した実装行に対してテストが存在するか確認する。カバーできていない行がある場合:
- **まず「このコードは本当に必要か」を問う**: 実運用上到達しないコードを発見した場合、カバレッジ対象外として扱ってはならない。削除・設計変更・`raise` への置き換えのいずれかを先に提案する（`tests/TestStrategy.md §7-A`）。
- 削除できない場合、「設計上このパスはテスト可能なはずでは？」と Phase 4 の設計に立ち返って確認する（外部境界フォールバックはテストを書ける — `tests/TestStrategy.md §7-B`）。
- テストを追加できる場合は実装してから再チェック。
- どうしても対処できない場合は `tests/TestStrategy.md §8 意図的未カバー領域` に類例がないか確認し、該当すれば理由欄に明記。非該当ならテスト不可の理由を説明してユーザーに承認を求める。

以下の表を必ずユーザーに提示し、承認を得てからコミットする:

```
## カバレッジ確認結果

### 追加テスト済み
- {ファイル:行} — {テストした内容}

### カバー不可（承認をお願いします）
| ファイル:行 | 理由 |
|---|---|
| {ファイル:行} | {理由（§8 該当行があれば明記）} |
```

> **⚠️ 重要**: カバレッジ確認結果の表は **PR 作成前**にユーザーに提示する必須ゲートである。
> テスト通過後にすぐコミット・PR 申請してはならない。カバレッジ確認 → 承認 → コミット の順序を守ること。

### コミット

実装完了・テスト通過・カバレッジ確認（承認済み）後に行う:
```bash
git add {変更ファイルを個別に指定}
git commit -m "{コミットメッセージ}"
```

コミットメッセージの形式: `{type}: {変更内容の要約} (closes #{番号})`
例: `feat: add pre-night judgment phase for day 2+ (closes #21)`

コミットメッセージの末尾に Co-authored-by タグを付ける。**実行しているモデル・ツールに合わせて**記述する（Claude Code か CODEX か、どのモデルかを正確に書く）:

```
# Claude Code（Anthropic）で実装した場合の例:
Co-Authored-By: Claude Sonnet 4.6 (1M context) <noreply@anthropic.com>
Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>

# CODEX（OpenAI）で実装した場合の例:
Co-Authored-By: OpenAI Codex <noreply@openai.com>
```

PR 本文末尾の `🤖 Generated with` 行も同様に実行環境に合わせて書く:

```
# Claude Code の場合:
🤖 Generated with [Claude Code](https://claude.com/claude-code)

# CODEX の場合:
🤖 Generated with [OpenAI Codex](https://openai.com/codex)
```

> 1. **AC 対応テスト確認表**（Phase 6「AC 対応テストチェック」）
> 2. **カバレッジ確認結果の表**（Phase 6「カバレッジ突合せ」）

PR の内容を以下のフォーマットで提示し、確認を取る:

本文:

## Summary
- {変更点1}
- {変更点2}

> **⚠️ PR 作成承認待ち中（およびフロー中の対話一般）**: ユーザーから「この条件要る？」「これって〇〇だよね？」のような確認の問いを受けたとき、コードや設計を変更する前に必ず**口頭で先に回答する**。回答 → ユーザーの意思確認 → 実装の順序を守る。

承認後、`mcp__plugin_github_github__create_pull_request` ツールで PR を作成する（長文 body をシェル経由で渡さない。現在ブランチは push 済みである必要がある）:

  owner="yukkie",
  repo="AgentVillage",

返り値の `url`（= PR URL）を表示する。

PR 作成後、**PR レビュー指摘**を受けたら、その指摘が当初 AC の範囲内かを確認する。当初 AC になかった要求が含まれる場合は下の「スコープ拡張ゲート（横断）」へ進む。

---

## スコープ拡張ゲート（横断）

任意のフェーズで発火しうる**割り込みゲート**。Phase 4.5（コードスメル対話）と同じく、特定フェーズに属さず必要なときに起動する。

### 発火条件

**当初 Issue の AC になかった要求**が外部から入ってきたとき。トリガーは実質この2つ:

1. **ユーザーの追加指摘**（対話中に新しい要求・観点が足された）
2. **レビュー指摘**（Phase 7 後、ユーザー・第三者・CODEX 等の PR レビューで AC 外の要求が判明）

> AI が実装中に自発的に「スコープが広がった」と気づくケースは想定しない（実態として起きない）。本ゲートは**外部入力に対するハンドラ**である。これを欠くと contract テストの設計が後追いになり、既存挙動との組み合わせ境界が漏れる（実例: #417 — 「CO 発言以降に継続表示」が後から追加され、「同一 day 内の CO 前発言には表示しない」×「前 day の CO が初期状態として継続表示される」の組み合わせ境界テストが抜けた）。

### 手順

発火したら、まず **追加/変更したい AC の案だけ** を提示する（この時点では Issue 編集も実装もしない）:

```
## スコープ拡張ゲート

### 追加/変更したい AC（案）
- [ ] {新 AC 1}
- [ ] {新 AC 2}

### 対処方針のサジェスト
| 観点 | 評価 |
|---|---|
| 追加 AC は独立した受け入れ条件として書けるか | {書ける / 元 AC と密結合} |
| 影響範囲（ファイル・レイヤ） | {元 Issue 内に閉じる / 別レイヤに波及} |
| 追加工数の目安 | {元 SP と同等以下 / 同等以上} |
| 元 AC との組み合わせ境界 | {疎 / 密} |

→ サジェスト: **{取り込む / 別 Issue 化 / 却下}**（理由: ...）

どうしますか？（取り込む / 別 Issue 化 / 却下 / AC 案を修正）
```

> **⚠️ 規模はユーザーが決める**: AI は「大きい/小さい」を勝手に断定して取り込みに流してはならない。
> 上表の客観シグナルを提示し、**最終判断はユーザーに委ねる**。

ユーザーの選択に応じて分岐する:

- **取り込む** — 下記 1→2→3 を順に行ってから、中断していたフェーズへ戻る
- **別 Issue 化** — `/issue create`（C-2/C-3）で新 Issue を作成する（別スキル。本フローからは `/issue create` の実行を案内する）。現 Issue は当初スコープで閉じ、追加分は新 Issue 側で別途実装する
- **却下** — スコープ外として扱い、当初 AC のまま進める（残す価値があれば `doc/Ideas.md` に追記）

「取り込む」を選んだ場合の手順（**この順序を守る**）:

1. **AC 更新** — 上記の AC 案で Issue 本文を更新する（`mcp__plugin_github_github__issue_write(method="update", owner="yukkie", repo="AgentVillage", issue_number={番号}, body="...")`）
2. **contract テスト設計** — 追加 AC に対応する contract テストを設計・追加する。`references/flow-*.md` の「Contract テストの TDD ルール」に従い、**RED を確認してから実装**する
3. **境界ケース列挙** — 追加 AC と既存テストの**組み合わせ境界**を列挙し、漏れを確認する（「新 AC が既存条件と交差する点」を洗い出す）

3 まで終えたら、中断元フェーズの「AC 対応テストチェック」へ戻り、更新後の AC で確認表を作り直す。

## Phase 8 — 教訓の振り返り（任意・pull 型）

PR 作成後の**区切り**に到達したら、`wisdom-capture` スキルを起動する。

このセッションで「ユーザーが Claude の提案を否認・修正し、その後に承認へ到達した」判断があれば、それは言語化できる暗黙知の候補。`wisdom-capture` がそれを検出し、ルール化（スキル追記）・Issue 化・寝かせる、を y/N で提案する。発火条件・動作・不変条件（主導権はユーザー・割り込まない・寝かせるを第一選択肢に）は `wisdom-capture/SKILL.md` に定義されているため、ここでは重複させない。

否認がなければ発火しないのが正しい。空振りを避けようと無理に教訓を捻り出さないこと。

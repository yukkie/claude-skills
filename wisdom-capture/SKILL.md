# Wisdom Capture — 教訓の言語化と固着

ベテランの判断には2種類ある。**(A) 暗黙の判断力**（無数の経験から圧縮された勘。言語化しきれない）と、**(B) 言語化できる方法論**（手順・判定基準に書き下せる）。このスキルは (B) を、作業のたびに静かに拾い上げ、監査可能・編集可能なテキストとして外部化する。

目的は「ユーザーを置き換える」ことではなく、**ユーザーを実行から解放して上流（何を言語化すべきか・どのルールが陳腐化したか・新しい方法論は何か）に専念させる**こと。だから主導権は常にユーザーに残す。

## いつ発火するか — 否認 → 承認到達

発火条件は2段階で、両方を満たしたときだけ起動する:

1. **否認シグナル**: セッション中に、ユーザーが Claude の提案・設計・案を**否認または修正**した（「それは違う、こうだ」「2分割じゃなく3分割」「その捉え方じゃなく、こう捉えろ」など）。否認は*暗黙知が表面化した瞬間*であり、最も確実な金脈。
2. **承認到達**: その否認の後、対話が進んで**ユーザーの承認に到達**した（修正後の案が受け入れられた／実装・Issue 化・PR まで通った）。承認到達は*その教訓が実際に正しいと確定した*ことを意味する。

**否認の瞬間には絶対に発火しない。** そこで「言語化しましょう」と割り込むのは、考えている最中のユーザーを遮る最悪の介入になる。承認に到達し、かつ**作業の区切り**（PR 作成後・タスク完了時・フェーズ完了時）に達したときに、一度だけ静かに差し出す。

否認がないセッション（無言の承認だけ）では発火しない。そこから暗黙知は拾えないため。

## 動作

### 1. 検出（内部）

このセッションで「否認 → 承認到達」を経た判断を拾う。各々について「**次回も再利用できる判定基準や手順の形に書き下せるか**」を確認する。書き下せるものだけが対象。その場限りの選択（この変数名はこっち、等）は除外する。

検出の質を上げる問い:
- ユーザーは何を否定し、代わりに何を主張したか
- その主張は「基準」や「手順」に一般化できるか（「この型を直す」ではなく「中心データ型を据えて伝播経路を列挙する」のように）
- それは*このプロジェクト固有*か、*汎用的な方法論*か（固着先の判断に効く）

### 2. ワンクッション確認

検出したら、作業の区切りで*静かに差し出す*。割り込まず、一度だけ:

```
いい教訓を得たのでルール化／Issue 化したいです。よいですか？ [y/N]

- {教訓1: 判定基準／手順の形に整形したもの}
- {教訓2: ...}
```

`N` なら何もしない。それで終わり。蒸し返さない。

### 3. 固着先の選択（y の場合のみ）

教訓ごとに行き先を選ぶ。**「今は固めず寝かせる」を常に第一選択肢として提示する。** 早すぎる言語化は、熟す前に摘んでしまい陳腐化・硬直を招く。勘がまだ転がっている段階の教訓は、固めない方が良いことが多い。

| 選択肢 | いつ | 行き先 |
|---|---|---|
| **寝かせる（既定）** | まだ熟していない・別定義の余地がある | 何もしない |
| **ルール化（スキル追記）** | 再利用する*方法論*。判定基準が安定している | 該当スキルの本文へ追記（汎用なら共通文書へ） |
| **Issue 化** | *作業*を伴う（ツール構想・改善タスク） | `/issue create` フロー、または直接 Issue 作成 |

汎用の方法論かプロジェクト固有かで、ルール化の追記先を選ぶ。固着先に迷ったら、それ自体をユーザーに聞く。

### 4. 移譲の可視化

ルール化・Issue 化したら、「今回 N 個をルール化／Issue 化しました（→ 追記先・Issue 番号）」と**明示する**。知識が静かに吸い上げられるのではなく、ユーザーが*移譲を意識的にコントロールできる*ようにする。怖さの大半は「気づかぬうちに」から来るため、移譲は常に見える化する。

## 不変条件

これらは設計の核であり、効率のために崩さない:

- **主導権はユーザー**。Claude は検出と提示までで、固着は必ず承認後。Claude が勝手に書き込まない。
- **割り込まない**。起動は否認→承認到達かつ区切りのみ。考えている途中で遮らない。否認の瞬間は特に厳禁。
- **寝かせるを尊重**。「今は言葉にしない」も正しい判断として常に第一に置く。
- **不発を恐れない**。否認がなければ発火しないのが正しい。空振りを避けようと無理に教訓を捻り出さない。

## なぜこの設計か

このスキルは「ベテランを再現可能にする」ことに成功するほど、知識を吸い上げる装置にも、有能な部下にもなりうる——同じ機能の裏表。解放装置の側に倒す鍵は、上の不変条件、とりわけ「主導権はユーザー」「割り込まない」「移譲の可視化」にある。知識を渡しても惜しくない人にとって、これは*教える手間を省く装置*になり、上流に登り続ける時間を生む。

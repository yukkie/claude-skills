# idd-team — マルチエージェント実装フロー

このスキルは AgentVillage の実装フロー（既存 Issue → 実装 → PR）を、**3つのロールに分業**して回す。

> **`/idd` との関係**: `/idd` は単一セッションで Claude が全フェーズを直接こなす。本スキルは同じフローを **オーケストレータ兼レビューア / 設計担当 / 実装・テスト担当** の3ロールに分け、設計担当・実装担当を Agent ツールでサブエージェントとして起動する。フローの規律（フェーズ・ゲート・事前調査・AC 照合）は `/idd` と同一であり、**本スキルはその規律を二重管理しない** — 各ロールに `/idd` SKILL.md と references を読ませて従わせる。本スキルが新しく定義するのは「**誰が何を担い、どう受け渡し、どこで人間に上げるか**」のオーケストレーション層だけ。

## なぜ分業するのか（役割の意図）

単一セッションだと、設計の思考・実装の手作業・全体のレビューがすべて同じコンテキストに混ざる。設計時の思い込みをそのまま実装に持ち込んだり、自分の実装を自分で甘くレビューしたりしやすい。ロールを分けると:

- **設計担当**は設計に集中し、その成果を Issue コメントという**検証可能な成果物**として残す。
- **実装・テスト担当**はその設計コメントを唯一の入力として実装する。設計の言語化が甘いとそもそも実装できないため、設計の質が強制的に上がる。
- **オーケストレータ兼レビューア**は自分で手を動かさず、各成果物を**第三者の目で検収**する。実装者が自分をレビューする利益相反が構造的に消える。

コールドスタートのサブエージェントでも仕事が繋がるのは、**状態を Issue コメントに書く**ことで「会話の流れ」を外部化しているから。これは AgentVillage が既に採っている「設計を Issue コメントに書いて流れを切れるようにする」運用と同じ思想。

## 3つのロール

| ロール | 担当フェーズ（/idd 基準） | 実体 | 入力 | 出力 |
|---|---|---|---|---|
| **オーケストレータ兼レビューア** | Phase 1-2、設計スキップ判定、全ゲート、検収、Phase 7-8 | このセッション（あなた） | ユーザー / 各サブエージェントの成果物 | ユーザーへの提示・承認取得・次ロールの起動 |
| **設計担当** | 事前調査・Phase 3-4（現状報告・設計提示・AC↔テスト対応） | Agent ツールのサブエージェント | Issue 番号・ブランチ名 | **Issue コメントに書いた設計案** |
| **実装・テスト担当** | Phase 5-6（doc 更新・実装・テスト・カバレッジ） | Agent ツールのサブエージェント | 承認済み設計コメントの URL | 実装差分・テスト・カバレッジ報告 |

## オーケストレーション全体像

```
[1] Issue 選択              ← オーケストレータ（ユーザーと対話）
[2] ブランチ作成            ← オーケストレータ（ユーザー確認）
        │
        ▼  AC を読んで設計フェーズの要否を判定
[S] 設計スキップ判定        ← オーケストレータ（スキップするなら理由説明 → ユーザー承認）  ★ユーザーゲート
     │                           軽量 Issue（バグ修正・文言・定数・doc のみ等）はここで設計担当を省略
     ├─ スキップ承認 → [I] 実装・テスト担当へ直行（設計担当を spawn しない）
     └─ スキップせず ↓
[P] 分割の一次判定          ← オーケストレータ（AC 数を数え、閾値超えなら spawn 前に分割提案）  ★ユーザーゲート（該当時）
     ├─ 分割する → /issue split を案内し中断
     └─ このまま → [D] へ
        │
        ▼  設計担当を spawn
[D] 設計担当: 事前調査 → 現状報告 → 設計案を Issue コメントへ投稿
        │
        ▼  オーケストレータが設計コメントを一次レビュー（検収）
[R1] 検収NG → 設計担当に差し戻し（再 spawn / 追加指示）
     検収OK → 設計案をユーザーに提示し承認を得る   ★ユーザーゲート
        │
        ▼  実装・テスト担当を spawn（入力＝承認済み設計コメント URL）
[I] 実装担当: doc 更新 → 実装 → テスト(RED→GREEN) → カバレッジ報告
        │
        ▼  オーケストレータが実装を一次レビュー（検収）
[R2] 実装担当から「設計差し戻し要求」→ ユーザーに上げ、設計し直すなら設計担当を再 spawn（崩れた前提を渡す）→ [D]→[R1] へ  ★ユーザーゲート
     検収NG → 実装担当に差し戻し
     検収OK → AC対応表・カバレッジ表をユーザーに提示       ★ユーザーゲート
        │
        ▼
[7] PR 作成                 ← オーケストレータ（ユーザー確認）  ★ユーザーゲート
[8] 教訓の振り返り（任意）   ← オーケストレータ

[*] スコープ拡張ゲート / コードスメル対話 — どのロールで顕在化しても、判断はオーケストレータがユーザーと行う
```

### 人間に上げるゲートと、自動で繋ぐ接続点

「オーケストレータが一次レビューして自動進行」が基本。ただし**不可逆・方向性を決める2点だけはユーザーに上げる**（既存 idd の哲学を維持し、差配が育つにつれ段階的に委譲できる余地を残す）:

- ★**設計スキップ判定**（[S]）— 設計フェーズを省くと判断したとき。誤って省くと設計の穴が実装まで届くため、理由を説明して承認を得る。
- ★**分割の一次判定**（[P]、該当時）— AC が閾値を超えたとき。設計担当を spawn する前に分割提案を上げ、無駄な設計やり直しを防ぐ。
- ★**設計承認**（[R1] 後）— 実装の方向を確定する。ここを誤ると全実装が無駄になる。
- ★**設計差し戻し**（[R2]）— 実装担当が承認済み設計の不備を発見したとき。設計し直すか判断するのはユーザー。設計担当の再起動を伴う。
- ★**PR 作成**（[7]）— 外向きの不可逆アクション。AC 対応表・カバレッジ表の承認も含む。

それ以外のフェーズ間ハンドオフ（事前調査→設計、設計→実装、doc更新→実装など）は、**オーケストレータがレビューアとして検収できれば人間を待たずに次へ進む**。検収で穴が見つかれば差し戻す（[R1]/[R2]）。コードスメル・スコープ拡張は外部入力ハンドラなので、発火したらフェーズに関わらずユーザーへ上げる。

> **差配が育つとは**: 最初はオーケストレータの検収が甘い前提で、ユーザーが設計承認ゲートで細かく見る。検収の精度が上がってきたら、ユーザーは設計承認を軽くし、最終的に PR 直前だけ見る運用へ寄せられる。スキル側で勝手にゲートを省略しない — ユーザーが「設計承認は任せる」と明言したときだけ緩める。

## Phase 1 — Issue 選択（オーケストレータ）

`/idd` の Phase 1 と同一。番号指定があれば Phase 2 へ。なければ `doc/Ideas.md` を読んで未実装 Issue 一覧を提示し、ユーザーの選択を待つ。Ideas.md にあるが GitHub で CLOSED の Issue の扱いも `/idd` Phase 1 に従う。

このフェーズはユーザーとの対話そのものなので、サブエージェントに渡さずオーケストレータが直接行う。

## Phase 2 — ブランチ作成（オーケストレータ）

`/idd` の Phase 2 と同一（オープン PR チェック → ブランチ名規約 → master ベースで作成）。git 操作とユーザー確認を伴うため、オーケストレータが直接行う。ブランチ名は次の設計担当・実装担当に渡すので確定させておく。

---

## [S] 設計スキップ判定（オーケストレータ）★ユーザーゲート

ブランチ作成後、設計担当を spawn する前に、**この Issue に設計フェーズ（設計担当の起動）が要るか**を判定する。これは `/idd` には無い idd-team 固有の分岐。マルチエージェントでは設計担当の spawn が1往復のコストを持つため、設計判断がほぼ不要な軽量 Issue ではそれを省いて実装担当に直行できると、ループ全体が目に見えて速くなる。

判定の前に対象 Issue の本文と AC を読む（`mcp__plugin_github_github__issue_read(method="get", ...)`）。

### スキップしてよい条件（いずれかに当てはまる）

設計書（doc）を読まなくても、既存実装を見れば正しく実装できる類の変更:

- **設計変更を伴わないバグ修正** — 既存の正しい設計に対する不具合の修正。新しい構造を導入しない。
- **新規関数追加を伴わない局所的な変更** — 既存関数の中身を直す・既存の呼び出しを差し替える程度で、新しい責務・抽象を作らない。
- **CSS など見栄えの変更** — スタイル・レイアウトの調整。挙動・データに触れない。
- **文言・コピー・i18n リソースの変更** — UI 文字列・翻訳リソースの修正。ロジックに触れない。
- **既存の確立したパターンへの単純追従** — 何度も繰り返されている書き方を1箇所増やすだけ（例: 既存定数を1つ追加、既存テストに1パターン足す）。
- **設定値・定数のチューニング** — タイムアウト値・ラウンド数など既存パラメータの値だけ変える。
- **ドキュメントのみの変更** — doc 修正だけで完結する。

### スキップしてはいけない除外条件（1つでも当てはまればスキップ不可）

一見小さくても設計判断が要る Issue。これらは「見かけの SP が小さいから設計不要」という罠の典型で、`/idd` の分割提案・schema drift 警告と同根:

- **中心契約型への変更を含む** — `LogEvent` / `EventType` / `ActorState` / 共通 JSON スキーマ / producer-consumer 境界モデルへの追加・変更。局所的に見えても schema drift の起点になる。データ契約 doc（`doc/DataSpec.md`）に触れる変更も同様。
- **横断的変更** — 複数モジュール・複数言語の consumer（CLI renderer / JS parser / replay / テスト）に波及する。
- **AC に独立変更項目が複数ある** — 分割提案（`/idd` の必須判定）の対象になりうる規模。
- **整理・構造化・refactor を求める Issue** — 対象そのものを変える設計判断が核心。スキップすると「対象を温存して新設する逃げ」を検収できない。

### 判定して提示する

スキップ条件に当てはまり、かつ除外条件に1つも当てはまらない場合のみ、スキップを提案する。**勝手にスキップして実装担当へ進めてはならない** — 理由を説明してユーザーの承認を得る:

```
## 設計スキップ判定 — Issue #{番号}

この Issue は設計フェーズ（設計担当の起動）を省いて実装担当に直行できると判断しました。

### 該当するスキップ条件
- {当てはまった条件}（例: 設計変更を伴わないバグ修正）

### 除外条件のチェック（すべて非該当を確認済み）
- 中心契約型への変更: なし
- 横断的変更: なし
- AC の独立変更項目: {N} 個（複数でない）
- 整理・構造化 Issue: 該当せず

### 根拠
{なぜ設計書を読まずに既存実装だけで正しく実装できるか、2-3文}

設計フェーズをスキップして実装に進んでよいですか？ [y/N]
（N の場合は通常どおり設計担当を起動します）
```

- **y** → 設計担当を spawn せず、**[I] 実装・テスト担当へ直行**する。ただし実装担当には依然として `/idd` の **事前調査の軽量版（producer/consumer の確認）と AC 照合は省かせない** — スキップするのは「設計担当による設計案の作成・設計承認ゲート」であって、AC を満たす実装とテストの担保ではない。実装担当プロンプトに「設計担当をスキップした軽量 Issue である。設計コメントは無いので、Issue 本文の AC を直接の仕様正本とする」と明記する。
- **N** → 設計をスキップせず [D] へ進む。

> **判断に迷ったらスキップしない**。設計担当を1回多く回すコストより、設計の穴が実装まで届いて手戻りするコストの方が高い。除外条件のどれかが「グレー」なら、それはスキップ不可のシグナル。

---

## [P] 分割の一次判定（オーケストレータ、設計担当を spawn する前）★ユーザーゲート（該当時）

設計をスキップしない場合、設計担当を spawn する**前に**、オーケストレータが分割提案の必須判定を行う。`/idd` ではこの判定は Phase 3（AC を読んだ直後）に置かれているが、idd-team では**設計担当の spawn が1往復のコストを持つ**ため、分割すべき Issue をそのまま設計担当に渡すと「設計させた後で分割が必要と分かって設計をやり直す」二度手間になる。AC のカウントは設計を要さない機械的作業なので、spawn 前にオーケストレータが先に済ませる（[S] 設計スキップ判定と同じ「spawn 前の軽量チェック」の思想）。

判定は `/idd` の「分割提案の必須判定」に従う — AC を読み、**独立変更項目を列挙してカウントする**。独立項目が **5 個以上**、または「再利用可能なコンポーネント／共通化の切り出し」が1件でも含まれる場合は、設計担当を spawn する前に分割提案をユーザーに提示する:

```
## 分割提案（AC 項目数 {N} 個）— 設計担当を起動する前に

この Issue の AC には独立した変更が {N} 個あります。このまま設計担当に渡して設計させると、
分割が必要だった場合に設計をやり直すことになります。先に分割方針を決めさせてください。

### 分割案
- **案A（最小遷移に絞る）**: {コア要件だけ本 Issue に残し、残りは別 Issue 化}
- **案B（重い切り出しを先行）**: {再利用コンポーネント等を先に別 Issue で実装してから本体}
- **案C（このまま1つで進める）**: {分割せず全項目を本 Issue で設計・実装する}

どうしますか？（案を選ぶ / このまま進む / 修正コメント）
```

- **分割する** → `/issue split <番号>` を案内する（子 Issue 作成＋依存リンクの双方向再配線を `issue split` が担保する。本フローから直接 `issue_write` で分割しない）。分割後、本フローは改めて Phase 1 から（または縮小したスコープで）やり直す。
- **このまま進む（案C）** → そのまま [D] へ。判定で「5 未満」だった場合もここに来る（提案を出さず黙って [D] へ進む）。

> **設計担当側でも二次確認はする**。オーケストレータの一次判定をすり抜けても、設計担当が事前調査の中で改めてカウントし、見落としがあれば設計コメントに「⚠️ 分割提案あり」として報告する（二重の網）。ただし**一次判定で捕まえれば設計担当の spawn を1回節約できる**のが [P] を置く価値。

---

## [D] 設計担当を起動

設計スキップせず、分割もしない（または分割後にスコープが確定した）場合、設計担当サブエージェントを Agent ツールで spawn する。設計担当は `/idd` の **事前調査 + Phase 3 + Phase 4** を実行し、成果を **Issue コメント** として残して返す。

サブエージェントへのプロンプトに必ず含める:

```
あなたは AgentVillage の「設計担当」サブエージェントです。

## 読むもの（この順に従う）
1. ~/.claude/skills/idd/SKILL.md の「事前調査」「Phase 3」「Phase 4」セクション
2. 変更対象の言語に応じた references/flow-python.md または flow-js.md の該当フェーズ
3. doc/CLAUDE.md（プロジェクト規約）

## タスク
- 対象 Issue: #{番号}（作業ブランチ: {ブランチ名}、チェックアウト済み）
- idd の事前調査（設計 Issue・横断的変更なら必須の全数洗い出し）→ Phase 3 現状報告 → Phase 4 設計提示 を、idd SKILL.md の指示どおりに行う。
- 分割提案はオーケストレータが spawn 前に一次判定済みだが、**二次確認として**あなたも事前調査の中で AC 独立項目を数え直す。オーケストレータの一次判定が見落としていた分割の必要（5 個以上 or 再利用コンポーネント切り出し）に気づいたら、設計コメントの冒頭に「⚠️ 分割提案あり（一次判定漏れ）」と明記して報告する。判断はユーザーが行うので、あなたは分割を実行しない。

## 重要: あなたはユーザーと直接対話できない
- 設計の選択肢・判断ポイントは「決め打ち」せず、Phase 4 の設計フォーマット（変更対象ファイル / 変更内容 / 設計上の判断ポイント / 影響範囲 / AC↔Contractテスト対応 / 関連Issue）を**そのまま** Issue コメントに書く。
- 判断ポイントは「A案 vs B案、推奨はAでその理由」の形で、オーケストレータがユーザーに提示できるよう選択肢を残す。
- コードスメル・設計レベルの問題（idd Phase 4.5 の対象）に気づいたら、設計コメント内に「## 設計上の問題点（要判断）」セクションを立てて列挙する。勝手に改善方針を確定しない。

## 出力（必ずこの形で返す）
1. mcp__plugin_github_github__add_issue_comment で対象 Issue に設計案コメントを投稿する。
   - コメント冒頭に `# [idd-team 設計案] #{番号}` の見出しを付ける（オーケストレータが識別するため）。
2. 投稿したコメントの URL と、設計の要約（3-5行）、分割提案・設計上の問題点の有無を、最終メッセージで返す。
- 実装・doc 更新・テストには着手しない。設計を Issue コメントに残すところまでがあなたの仕事。
```

> **なぜ Issue コメントに書かせるか**: 実装・テスト担当はコールドスタートで、この設計の思考過程を共有していない。設計を Issue コメントという永続物に外部化することで、実装担当が（さらには後から人間や別セッションが）設計の流れを完全に辿れる。サブエージェントの最終メッセージだけでは消えてしまう。

## [R1] 設計の検収 → ユーザー承認（オーケストレータ）

設計担当が返したら、オーケストレータが**レビューアとして一次検収**する。投稿された設計コメントを `mcp__plugin_github_github__issue_read(method="get_comments", ...)` で読み、以下を確認する:

- 事前調査が必要な Issue（設計 Issue・横断的変更）で、producer/consumer の全列挙・同種ハック探索が実際に行われているか。grep が届かない境界（idd スキル定義など別リポジトリの consumer）への言及があるか。
- AC が**全項目転記**され、各 AC に対応する contract テストが設計されているか。「contract テスト不可」の AC には理由と代替手段があるか。
- 設計が Issue の目的からドリフトしていないか（整理 Issue なのに対象を温存して新設する逃げになっていないか — idd Phase 4 のドリフト確認）。
- 判断ポイントが選択肢の形で残されているか（決め打ちで潰されていないか）。

**検収NG** → 設計担当を再 spawn するか、追加指示を与えて穴を埋めさせる（[D] に戻る）。何が足りないかを具体的に伝える。

**検収OK** → 設計コメントの内容をユーザーに提示し、承認を得る ★**ユーザーゲート**:

```
## 設計担当の設計案（検収済み）

{設計コメントの要約。判断ポイントは選択肢のまま提示}

設計コメント全文: {Issue コメント URL}

※ 分割提案 / 設計上の問題点があればここで併せて提示し、ユーザーの判断を仰ぐ

この設計で実装に進んでよいですか？ [y/N / 修正コメント]
```

- **分割提案あり** → idd の規定どおり分割案を提示。分割するなら `/issue split <番号>` を案内し、本フローは中断する。
- **設計上の問題点あり**（コードスメル等） → idd Phase 4.5 に相当。ユーザーと対処方針（改善/後回し/このまま）を決めてから進む。改善が必要なら設計担当を再 spawn。
- **修正コメント** → 設計担当に修正指示を出して再 spawn。
- **AC 追従が必要**（実装方針が AC の表記とズレる設計） → idd の「AC 追従チェック」に従い、ユーザー承認のうえ Issue 本文の AC を更新してから進む。
- **y** → 実装・テスト担当の起動へ。

> 設計承認は外部レビュー（idd の「設計承認後の外部レビュー」）の機能を兼ねる — 設計は既に Issue コメントとして可視化されているため、ユーザーや第三者・別 AI がそのまま読める。

## [I] 実装・テスト担当を起動

実装・テスト担当サブエージェントを spawn する。実装担当は `/idd` の **Phase 5 + Phase 6** を実行する。入力は経路によって変わる:

- **[D] 経由（通常）**: 入力は**承認済み設計コメントの URL**。これが設計仕様の正本。
- **[S] スキップ経由（軽量 Issue）**: 設計コメントは存在しない。**Issue 本文の AC が直接の仕様正本**。下のプロンプトの「読むもの 1」をコメント URL から「Issue 本文（特に AC）」に差し替え、「設計担当をスキップした軽量 Issue である」と明記する。

サブエージェントへのプロンプトに必ず含める:

```
あなたは AgentVillage の「実装・テスト担当」サブエージェントです。

## 読むもの（この順に従う）
1. 承認済み設計コメント: {Issue コメント URL} ← これがあなたの設計仕様の正本
2. ~/.claude/skills/idd/SKILL.md の「Phase 5」「Phase 6」セクション
3. 変更対象の言語に応じた references/flow-python.md または flow-js.md の該当フェーズ
4. doc/CLAUDE.md（プロジェクト規約。絶対 import・コミット規約など）

## タスク
- 対象 Issue: #{番号}（作業ブランチ: {ブランチ名}、チェックアウト済み）
- 承認済み設計コメントに従って Phase 5（doc 更新）→ Phase 6（実装・テスト・カバレッジ）を idd SKILL.md の指示どおりに行う。
- Contract テストは idd の TDD ルールに従い、**アサートレベルの RED を確認してから実装**する。機能削除の退行テストは GREEN 後に削除する。
- テスト docstring の4要素（SUT/Mock/Level/Objective）を揃える。
- 実装後チェック（lint・test・カバレッジ、JS なら Playwright 動作確認）を references に従って実行する。

## 重要: あなたはユーザーとも設計担当とも直接対話できない
- 設計から逸脱したくなったら、勝手に判断せず最終メッセージで「設計と実装の乖離」として報告する。コミット・PR には進まない。
- **承認済み設計そのものが不十分・実装不能・誤りだと判明したら、設計の穴を力技で埋めず「設計差し戻し要求」として報告する**（idd SKILL.md「設計フェーズへの差し戻し」を参照）。例: 設計が前提したデータ構造/関数が存在しない、設計が見落とした consumer/producer が見つかった、AC を満たすには別アプローチが要ると分かった。あなたは設計担当に直接戻せないので、何が・なぜ設計どおり実装できないかを具体的に書いて最終メッセージで返す（オーケストレータが設計担当を再起動する）。
- 実装中に当初 AC になかった要求に気づいても（実態としては稀）、スコープ拡張はユーザー判断事項なので、実装せず報告に留める。
- 実装中に既存コードとの一貫性向上・シンプル化に気づいたら、適用せず最終メッセージで提案として挙げる。

## 出力（必ずこの形で返す）
1. 実装・テストを完了し、lint/test/カバレッジを通す（コミットはまだしない）。
2. 最終メッセージで以下を返す:
   - 変更ファイル一覧
   - AC 対応テストチェック表（AC 全項目転記 → 対応実装ファイル:行 → 対応テスト関数名 → 状態）
   - カバレッジ確認結果（追加テスト済み / カバー不可とその理由）
   - RED→GREEN を確認した contract テストの一覧
   - 設計からの乖離・要判断事項・改善提案があれば明記
   - **設計差し戻し要求があれば最優先で明記**（設計の崩れた前提・影響・なぜ実装で埋められないか）。この場合は実装を途中で止めてよい。
- コミット・PR 作成はしない（オーケストレータがユーザー承認後に行う）。
```

> **なぜコミットさせないか**: AC 対応表とカバレッジ表は idd で「PR 作成前の必須ユーザーゲート」と定められている。実装担当がコミットまで進むとこのゲートを飛ばすことになる。コミット権はオーケストレータが握り、ユーザー承認後に行う。

## [R2] 実装の検収 → ユーザー承認（オーケストレータ）

実装担当が返したら、まず**設計差し戻し要求が含まれていないか**を最初に確認する。

### 設計差し戻し要求を受けた場合（最優先ハンドラ）

実装担当が「承認済み設計では実装できない」と報告してきたら、検収・コミットに進む前にこれを処理する。実装担当はユーザーにも設計担当にも直接戻せないため、**この差し戻しを受けて次の手を打つのはオーケストレータの責務**:

1. 差し戻し要求の内容（崩れた設計の前提・影響・なぜ実装で埋められないか）を確認する。
2. ユーザーに提示する ★**ユーザーゲート**（idd「設計フェーズへの差し戻し」に相当）:
   ```
   ## 設計フェーズへの差し戻し要求（実装担当より）

   実装担当が、承認済み設計では対応できない点を実装中に発見しました。

   ### 設計の崩れた前提
   - {設計が前提していたこと} → {実際に判明したこと}

   ### 影響
   - {このまま実装を続けると何が問題か}

   設計を見直しますか？（設計し直す / 設計を一部修正して続行 / このまま進める理由がある）
   ```
3. ユーザーが「設計し直す/一部修正」を選んだら、**設計担当を再 spawn する**（[D] へ戻る）。このとき再 spawn プロンプトに、実装担当が発見した「崩れた前提」を**追加入力として渡す** — 設計担当はコールドスタートなので、何が問題で差し戻されたかを知らないと同じ設計を返してしまう。設計担当が設計コメントを更新したら、再び [R1]（設計承認ゲート）を通してから実装担当を再 spawn する。
4. ユーザーが「このまま進める理由がある」と判断したら、その方針で実装担当を再 spawn する。

> **力技で握りつぶさせない**: 実装担当が設計の穴を自力で埋めて実装を完成させてきた場合（差し戻すべきだったのにしなかった場合）も、検収で「設計コメントと実装の乖離」として検出される。乖離が設計のミス起因なら、完成していても差し戻しハンドラに回す。設計と実装の整合（検証可能性）を優先する。

### 通常の検収

差し戻し要求がなければ、オーケストレータが**レビューアとして検収**する:

- AC 対応表が**全項目**埋まっているか。「対応実装」列が空欄の AC（＝未実装）がないか。照合の向きが「AC → 実装/テスト」になっているか（「書いたコード → テスト」で実装漏れ AC を見落としていないか）。
- contract テストが**アサートレベルの RED**を経ているか（コンパイルエラー RED の false positive でないか）。
- カバレッジのカバー不可項目に正当な理由（TestStrategy.md §8 等）があるか。実運用上到達しないコードを「カバレッジ対象外」と誤魔化していないか。
- 設計担当の設計コメントと実装が一致しているか。乖離報告があれば内容を確認する。

**検収NG** → 実装担当を再 spawn して穴を埋めさせる（[I] に戻る）。

**検収OK** → AC 対応表・カバレッジ表をユーザーに提示し承認を得る ★**ユーザーゲート**（idd Phase 6 の必須ゲートそのもの）。承認後、オーケストレータがコミットする（idd のコミット規約・Co-authored-by に従う。実体は実装担当が書いたコードなので、コミットメッセージにその旨を含めてよい）。

> **乖離・改善提案・スコープ拡張の報告があった場合**: コードスメル/スコープ拡張ゲートと同様、ユーザーに上げて判断を仰ぐ。「改善提案を取り込む」なら実装担当を再 spawn。「スコープ拡張を取り込む」なら idd のスコープ拡張ゲート手順（AC 更新 → contract テスト RED → 境界ケース列挙）を踏ませる。

## Phase 7 — PR 作成（オーケストレータ）★ユーザーゲート

`/idd` の Phase 7 と同一。前提チェック（AC 対応表・カバレッジ表がともにユーザー承認済み）を満たしたうえで、Ideas.md の該当行削除 → PR 内容提示 → ユーザー承認 → `create_pull_request` の順に進む。git 操作・外向きアクションのためオーケストレータが直接行う。

PR レビュー指摘で当初 AC 外の要求が来たら、スコープ拡張ゲート（横断）へ。

## Phase 8 — 教訓の振り返り（任意・pull 型）

`/idd` の Phase 8 と同一。PR 作成後の区切りで `wisdom-capture` を起動する。本スキル特有の観点として、**オーケストレーションの分担・受け渡しで詰まった点**（設計コメントの粒度が粗くて実装担当が手戻りした、など）があれば、それは idd-team スキル自体の改善候補になりうる。

## 横断ゲート（どのロールで顕在化しても判断はオーケストレータ）

- **コードスメル対話**（idd Phase 4.5）: 設計担当が設計コメントに「設計上の問題点」を挙げる、または実装担当が乖離報告で挙げる。オーケストレータがユーザーと対処方針を決め、必要なら該当ロールを再 spawn。
- **スコープ拡張ゲート**（idd 横断）: ユーザーの追加指摘 / PR レビュー指摘で当初 AC 外の要求が来たとき。AI が自発的に「広がった」と気づくケースは想定しない（外部入力ハンドラ）。オーケストレータが AC 案を提示し、取り込む/別 Issue 化/却下をユーザーに選ばせる。「取り込む」なら実装担当に AC 更新→RED→境界ケースを踏ませる。

これらの手順詳細は `/idd` SKILL.md に定義済み。本スキルでは**起動主体がオーケストレータである**ことだけを上書きする。

## サブエージェント運用の注意

- **同じブランチを共有する**: 設計担当・実装担当・オーケストレータは同一作業ブランチ上で動く。設計担当はコードを変えない（Issue コメントのみ）ので競合しない。実装担当が差分を作り、オーケストレータがコミットする。
- **コールドスタート前提で渡す**: サブエージェントは会話履歴を持たない。Issue 番号・ブランチ名・設計コメント URL・読むべきファイルパスを毎回明示的に渡す。「さっき決めた」は通じない。
- **検収を省略しない**: オーケストレータが手を動かさない代わり、検収がこのスキルの品質保証の要。サブエージェントの最終メッセージを鵜呑みにせず、Issue コメント・実際の差分・テスト結果を自分で読んで確認する。
- **再 spawn のコスト**: サブエージェントは毎回コールドスタートで起動コストがある。差し戻しは「何が足りないか」を具体的に伝え、一度の再 spawn で埋まるようにする。
- **モデルはオーケストレータが Issue の性質に応じて選ぶ**: Agent ツールの `model` を省略すると親（オーケストレータ）から継承され、明示すればオーバーライドできる。固定値は定めない — オーケストレータが Issue の難易度・性質を見て選ぶ。判断材料:
  - **設計担当**は事前調査の網羅性・複数案のトレードオフ判断が成果を左右するので、設計 Issue・横断的変更・中心契約型が絡むものほど賢いモデル（opus 等）を充てる価値が高い。
  - **実装・テスト担当**は設計が確定した後の実装・TDD・カバレッジが主で、軽量〜中規模なら速いモデル（sonnet 等）で足りることが多い。設計差し戻しが起きやすそうな難所では引き上げる。
  - 迷ったら継承（親と同じ）で始め、検収で質が足りなければ次の spawn でモデルを上げる。`subagent_type` は基本 `general-purpose`（プロジェクト固有のカスタムエージェントがあればそれでもよい）。

## エラー処理

| 状況 | 対応 |
|---|---|
| 設計担当が Issue コメントを投稿せず最終メッセージだけ返した | コメント投稿を必須要件として再 spawn。流れの外部化が崩れるため妥協しない |
| 実装担当がコミット・PR まで進んでしまった | ユーザーゲート（AC/カバレッジ表）を飛ばしているため、コミットを確認しユーザー提示に戻す |
| 設計コメントと実装が食い違う | 設計担当の設計が正本。乖離が設計のミスなら設計担当を、実装のミスなら実装担当を再 spawn |
| サブエージェントが idd の規律を守っていない（事前調査スキップ等） | 該当 idd セクションを明示して再 spawn。本スキルは idd の規律を二重定義しないため、必ず idd を読ませる |
| その他（issue_read 失敗・ruff/pytest 失敗・ブランチ既存） | `/idd` のエラー処理表に従う |

## 将来の発展メモ — カスタム Agent 定義への移行（未着手）

現在、設計担当・実装担当は `subagent_type: general-purpose` に長いプロンプト（[D]/[I] のブロック）でロールを説明して spawn している。これを**カスタム Agent 定義**（例: `design-agent` / `implement-agent`）に移すと、以下が改善できる:

- **本文の簡素化**: [D]/[I] の「あなたは〇〇担当です…」ブロックを定義側に常駐させ、本スキルは「定義を Issue 番号・ブランチ名・読むファイル・（設計担当なら不要／実装担当なら設計コメント URL）を添えて spawn する」だけにできる。
- **ツール制限による構造的保証**: 設計担当の定義から Edit/Write を外せば、「設計担当はコードに触らず Issue コメントだけ返す」を**プロンプトの遵守頼みでなく物理的に**担保できる（「設計担当が勝手に実装し始める」事故が定義レベルで不可能になる）。
- **モデルのデフォルト焼き込み**: 設計担当=opus・実装担当=sonnet を定義のデフォルトにできる（オーケストレータは Issue の性質で必要時のみオーバーライド）。

**移行時の不変条件**: 本スキルの設計思想「フローの規律を二重定義しない（idd SKILL.md を読ませる）」を崩さないこと。Agent 定義には**ロールの性格・ツール制限・デフォルトモデルだけ**を置き、事前調査・AC 照合・TDD などフローの規律は引き続き idd を読ませて従わせる。規律を定義にも書くと idd / idd-team / Agent 定義の3箇所に分散する。

**進め方の推奨**: いきなり定義前提に本文を書き換えず、まず `design-agent` を1つ作って挙動を確かめてから（Edit/Write を外した状態で設計コメントを正しく返すか）、良ければ `implement-agent` も作り、本文を定義前提に簡素化する。評価が固まってから着手する。
